# AgroDoc V2.0: статус внедрения и ориентация

Краткий документ для быстрого понимания: что задумывалось в исходном плане фундамента, что реализовано, что добавлено сверх плана, куда смотреть в репозитории. Без деталей кода, только смысловые опоры.

---

## 1. Исходный план (о чём речь)

Под «планом фундамента» имелся в виду единый контур V2: монорепозиторий, бэкенд с доменной логикой, PostgreSQL со схемой под заявки и сопутствующие сущности, движение по статусам с историей, аудит и события, хранение файлов, очереди RabbitMQ и воркеры Celery, два веб-интерфейса (приложение и админка) плюс общий UI, контейнеризация и сценарий локального запуска, справочники (lookups) и границы прав для административных операций.

---

## 2. Что из плана сделано (по блокам)

**Репозиторий и инфраструктура:** структура монорепозитория, Docker Compose (PostgreSQL, RabbitMQ, бэкенд, несколько воркеров Celery, две фронтенд-сборки), Makefile, CI на GitHub (сборка образов, бэкенд-джоб с Postgres и миграциями, тесты, отдельный сценарий e2e по API). Образ бэкенда при старте прогоняет миграции Alembic, затем поднимает API.

**База и миграции:** заявки, ревизии, ссылка на текущую ревизию, история смены статусов, журнал аудита с типом события, файлы с хешем, таблица доменных событий, индексы, учётные записи и справочники (роли, статусы и т.д.), согласованные с ORM-моделями. Отдельная миграция вводит тестовую учётку для CI и e2e (логин в документации к API и в README, не для production).

**Бизнес-логика заявок:** создание заявки и ревизий, смена статуса только через сервис, матрица переходов по ролям, запись в историю и аудит, постановка фоновой обработки события после создания/смены статуса.

**Слой событий и фоновые задачи:** регистрация обработчиков по типу события, очереди Celery под документы, уведомления, события и обслуживание, задача обработки доменного события с попыткой вызвать внешние воркеры (в тестах мокается, чтобы не требовать брокер).

**Файлы и хранилище:** загрузка с привязкой к сущности, локальное хранилище по корню на диске, вычисление контрольной суммы, опциональная схема S3-совместимого хранилища и выдача скачивания по тому же абстрактному бэкенду. Уведомление о загрузке через Celery (по возможности).

**Справочники и админ-контур API:** чтение кодов ролей, статусов, типов источника и файлов, создание новой роли в справочнике с проверкой прав (админ), запись в аудит; **админ-учётки:** список, создание, `PATCH`, сброс пароля (`/admin/accounts…`).

**Публичные границы API заявок:** просмотр заявки, пагинированный список с фильтрами по статусу и дате создания, выдача списка ревизий и записей аудита по заявке, создание с учётом авторизованного пользователя где применимо.

**Безопасность и идентичность:** пара логин-пароль, **access** и **refresh** JWT (`typ`, `jti` в refresh), серверные строки в **`auth_refresh_sessions`** с отзывом при refresh-ротации, **`POST /auth/logout`**, смена своего пароля **`POST /auth/change-password`** с отзывом всех refresh-сессий учётки; при смене пароля/роли/деактивации админом — массовый отзыв refresh для затронутой учётки. Проверка паролей bcrypt, профиль по access, отладочный обход заголовками для тестов.

**Тесты и приёмка:** unit/API-тесты на критичные пути, отчёт e2e через Playwright (здоровье API, логин, опционально лендинги приложения и админки если сервисы на портах доступны).

**Документация в репо:** корневой README с портами, тестами, e2e, API-списком, хранилищем и переменными окружения, пример env-файла, этот документ, краткие инфраструктурные примечания.

**Фронтенды (приложение и админка, React):** `shared-ui`: HTTP-клиент (Bearer + **refresh**, **тихий refresh** при 401, **`logoutRemote`** перед очисткой storage), `readApiError`. Контекст авторизации, вход, защищённые маршруты, страница **«Пароль»** (самообслуживание). Приложение: список заявок, **создание заявки**, карточка (**смена статуса** с подсказкой целей с API, ревизии с диффом, аудит, файлы). Админка: панель, справочники, **«Пользователи»**, **«Аудит»**. Сборка образов: `shared-ui/src`, `VITE_API_BASE_URL`.

## 3. Дополнительно к исходному плану (сверх базового фундамента)

Сводка вещей, которые явно вышли за рамки «только фундамента» в узком смысле, но вошли в текущую поставку:

- Полноценный контур **JWT-авторизации** вместо долгой жизни на заглушках заголовков: логин, access+refresh, «кто я», silent renew, согласованность с учёткой и ролью.
- **Сиды под автотесты** (отдельная миграция) и явное описание тестовой учётки в README для локальной/CI-отладки, с напоминанием не использовать то же в бою.
- **Расширенный публичный API чтения**: список с пагинацией и фильтрами, ревизии и аудит в одном стиле с остальными маршрутами.
- **Скачивание файлов** и единая абстракция чтения для локального диска и S3-URI.
- **Упаковка backend-образа** с автоматическим `alembic upgrade` перед Uvicorn, чтобы подъём стека в Compose не требовал ручного шага миграции для «просто запусти».
- **Расширенный CI**: e2e-слой против живого API (Playwright), плюс переменные окружения, согласованные с бэкендом.
- **Каталог e2e** в корне: зависимости, конфиг Playwright, сценарии API и опциональные UI, package-lock для воспроизводимой установки.
- **Интеграция фронтов с API** (клиент, сессия, UI списка и деталей) как осознанно расширенный слой поверх «заглушек страниц».

## 4. Где что искать (ориентиры по путям, без кода)

- **Бэкенд приложения и роуты:** каталог `backend/app`, в том числе `api` (маршруты), `core` (настройки, крипто, токены), `services` (сценарии, хранилище, аудит, workflow), `models`, `migrations`.
- **Сборка и тесты бэкенда:** `backend/Dockerfile`, `backend/pytest.ini`, `backend/tests`, `backend/requirements.txt`.
- **Фронтенды и общий стиль:** `frontend-app`, `frontend-admin`, `shared-ui` (в т.ч. `shared-ui/src` — общий API-клиент, подключается в оба приложения).
- **Состояние входа на фронте:** `auth` (контекст), `lib/api.ts` (обёртка над клиентом), `RequireAuth`, страницы `Login*`.
- **E2E:** `e2e` (тесты и конфиг Playwright).
- **Состав стека в Compose:** `docker-compose.yml` в корне, переменные — из `.env.local` по образцу `.env.example`.
- **Служебные и инфраструктурные заметки (маршрутизация, порты):** `infra/README.md`.
- **CI:** `.github/workflows`.

## 5. Что логично считать следующим этапом (не в текущем «сделано»)

- **UX:** доработка форм (валидация до отправки, единые баннеры успеха/ошибки), карточка заявки — пагинация таблиц, экспорт.
- **Учётки:** редактирование ФИО/email в UI; опционально чёрный список access JWT или ротация `JWT_SECRET_KEY` для полного сброса.
- **Политика прав на скачивание** и ограничение по `counterparty`/участнику, если потребуется.
- **E2E:** больше сценариев после входа (карточка заявки, создание учётки в UI).
- **S3 в бою** (секреты, бакет-политики) и **описание воркеров Celery** для эксплуатации.
